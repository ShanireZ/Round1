# 决策治理规范

## 目标

让 Round1 在项目后半程保持一致的技术、设计和运营口径，避免旧计划、新实现、agent 产出、临时脚本互相漂移。

治理标准的目标是降低真实风险，不是增加审批仪式。新增规则必须能回答三个问题：保护什么不变量、由谁在什么时候检查、失败时如何恢复或收口。

## 标准设计原则

- 规则优先保护安全、隐私、数据完整性、考试公平、内容可追溯和已定 UI/UX。
- 每条“必须/禁止”应能被测试、review、脚本、截图、runbook、审计记录或发布记录验证。
- 不把大厂实践照搬成流程；只吸收清晰职责、小步变更、可观测、可回滚、持续改进等方法。
- 默认使用轻量记录：PR 描述、`docs/plans/*`、reference 更新即可；只有 S0/S1 需要更完整决策记录。
- 标准应允许合理例外，但例外必须有到期条件，不能变成永久口子。

## 决策分级

| 级别 | 示例 | 要求 |
| --- | --- | --- |
| S0 不可破坏 | 认证安全、考试公平性、数据不可逆迁移、线上删除数据 | 必须有计划、测试、回滚或恢复方案；至少一人复核 |
| S1 架构决策 | prebuilt-only、生产 no-runner、LLM provider-direct、配置真源 | 必须更新 `plan/reference-*` 与本目录相关规范 |
| S2 产品体验 | 页面 IA、UI token、核心流程、文案语气 | 必须遵循 UI/UX 定稿，不得重新设计 |
| S3 局部实现 | 单个组件、单个路由、脚本参数 | 可在 PR 中说明，不必单独 ADR |

## 决策证据

不同级别需要的证据不同，避免低风险变更被重流程拖慢，也避免高风险变更只靠口头判断：

| 级别 | 最小证据 | 验收重点 |
| --- | --- | --- |
| S0 | 计划/ADR、回滚或恢复路径、针对性测试、复核记录 | 是否能避免或恢复用户/数据/权限事故 |
| S1 | reference/standard 更新、兼容策略、验证命令 | 是否维持长期架构口径 |
| S2 | UI/UX 或业务流程对照、截图/手工验收 | 是否符合已定体验与核心流程 |
| S3 | PR 描述、局部测试或说明 | 是否清楚、可维护、无越界影响 |

证据不要求固定模板，但必须让后来的维护者能复盘“为什么这样做”。

## 单一真源

- UI/UX 真源：`plan/uiux_plan.md` + [04-ui-ux.md](04-ui-ux.md)。
- 环境变量代码真源：`config/env.ts`。
- LLM 路由真源：`config/llm.ts`。
- 数据模型真源：`server/db/schema/**`、迁移文件与 `plan/reference-schema.md`。
- API 现状真源：`server/app.ts`、`server/routes/**`、OpenAPI registry、`plan/reference-api.md`。
- 运行时配置优先级：`app_settings > .env > 代码默认值`，目标契约见 `plan/reference-schema.md`。
- 业务状态机真源：`plan/step-*`、`plan/reference-schema.md` 与 [20-product-state-and-workflow.md](20-product-state-and-workflow.md)。
- 隐私与数据生命周期真源：[21-privacy-and-data-lifecycle.md](21-privacy-and-data-lifecycle.md)。
- 计划状态真源：较新的 `plan/step-*`、`plan/reference-*`、`docs/plans/*` 对齐说明。

## 变更流程

1. **识别影响面**：至少标注是否影响 UI、API、数据库、权限、配置、部署、离线内容、agent 调用。
2. **确定依据**：引用本目录和对应 plan/reference 文档。
3. **小步修改**：每次变更保持可审查；避免把样式重设、业务重构和迁移揉在一个 PR。
4. **验证闭环**：按 [11-testing-quality.md](11-testing-quality.md) 完成分层测试。
5. **同步文档**：改了用户可见行为、部署、配置、脚本、状态机、内容产物命名，就必须同步文档。

变更拆分以“可独立验证”为准，而不是机械按文件类型拆分。一次迁移可以配套必要代码和测试，但不应顺手夹带无关 UI polish 或命名重构。

## 例外机制

任何例外必须写入 PR 描述或计划文档：

- 例外规则编号或文件名。
- 为什么现在不能遵守。
- 风险范围与用户影响。
- 临时防护措施。
- 后续收口日期或触发条件。

禁止使用“时间不够”“先这样”作为唯一理由。

## 冲突处理

- 新规范与 UI/UX 定稿冲突：保留 UI/UX 定稿，先修改规范。
- 新需求与 prebuilt-only 冲突：需求必须重写，不得恢复在线组卷、在线换题或线上内容生成。
- 新实现与已有测试冲突：先判断测试是否仍代表目标契约；如是，改实现；如否，先改计划与测试。
- 文档之间冲突：以更新时间更晚且明确写有“当前对齐说明”的文档为准。
- 外部规范与 Round1 当前约束冲突：保留 Round1 约束；如要采纳外部做法，先写成本、收益、迁移路径。

## 评审原则

采用 Google 工程实践中“持续改善代码健康”的思路：评审不追求个人偏好，优先看设计合理性、用户影响、复杂度、测试、命名、注释、文档和一致性。纯样式偏好不得覆盖本目录和 UI/UX 定稿。

## 决策记录模板

触碰 S0/S1 的变更必须在 `docs/plans/` 或 ADR 中写清：

```markdown
# YYYY-MM-DD Topic

## Context

## Decision

## Alternatives Considered

## Impact

## Migration Plan

## Verification

## Rollback
```

## Round1 不变量

以下不变量在普通需求中不得被推翻：

- 生产运行时不生成题、不跑 cpp-runner、不在线换题。
- UI/UX 定稿不被单页面重设。
- Admin 敏感操作必须 step-up 和审计。
- 已发布预制卷不可原地覆盖。
- 可导入内容资产必须有 `runId`、checksum 和 import batch。
- `config/env.ts`、`config/llm.ts`、DB schema、OpenAPI registry 是代码真源。
- 状态迁移必须由服务端完成，并满足 [20-product-state-and-workflow.md](20-product-state-and-workflow.md) 的幂等、审计和不变量。

## 风险接受

允许接受低风险残余问题，但必须显式记录：

- 残余风险具体是什么，不写“有风险”这类空话。
- 为什么当前阶段可以接受。
- 通过什么监控、回滚、人工 review、feature flag 或后续任务降低风险。
- 什么事件会触发重新处理，例如首次线上回归、同类 bug 第二次出现、发布前演练失败。

不得接受会导致越权、答案丢失、不可逆数据破坏、隐私泄露或已发布内容资产失真的风险。

## 角色职责

| 角色 | 职责 |
| --- | --- |
| Feature owner | 明确目标、风险、验收和文档同步 |
| Implementer | 小步实现、补测试、维护现有约束 |
| Reviewer | 审查安全、数据、状态机、UI/UX、测试 |
| Release operator | 执行部署、备份、健康检查、回滚准备 |
| Content operator | 生产、校验、导入、发布离线内容资产 |

AI agent 执行任务时同时承担 implementer 职责，但不能替代 reviewer 的独立判断。

## 完成定义

一项任务只有同时满足以下条件才算完成：

- 代码或文档已落地。
- 相关测试/验证已运行并记录。
- 计划/reference/标准中被影响部分已同步。
- 已说明未覆盖风险。
- 没有引入未授权的架构、UI、数据或安全例外。

## 例外到期

例外不得永久存在。每个例外必须有：

- 到期日期，或
- 触发收口的事件，例如“Coach API 挂载后删除兼容路径”，或
- 明确 legacy 标记和迁移计划。

## 文档漂移处理

发现 plan、standard、代码、测试冲突时：

1. 先确定当前代码是否代表目标状态。
2. 如果代码是临时兼容，文档应写“当前/目标”双口径。
3. 如果代码已代表新目标，更新旧计划并追加状态说明。
4. 如果无法判断，暂停扩大修改，补调查记录。

漂移处理优先修正最容易误导执行者的入口：README、reference、执行计划、标准索引。历史计划可以保留，但顶部必须指向当前真源或标记为历史记录。
