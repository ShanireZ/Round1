# 决策治理规范

## 目标

让 Round1 在项目后半程保持一致的技术、设计和运营口径，避免旧计划、新实现、agent 产出、临时脚本互相漂移。

## 决策分级

| 级别 | 示例 | 要求 |
| --- | --- | --- |
| S0 不可破坏 | 认证安全、考试公平性、数据不可逆迁移、线上删除数据 | 必须有计划、测试、回滚或恢复方案；至少一人复核 |
| S1 架构决策 | prebuilt-only、生产 no-runner、LLM provider-direct、配置真源 | 必须更新 `plan/reference-*` 与本目录相关规范 |
| S2 产品体验 | 页面 IA、UI token、核心流程、文案语气 | 必须遵循 UI/UX 定稿，不得重新设计 |
| S3 局部实现 | 单个组件、单个路由、脚本参数 | 可在 PR 中说明，不必单独 ADR |

## 单一真源

- UI/UX 真源：`plan/uiux_plan.md` + [04-ui-ux.md](04-ui-ux.md)。
- 环境变量代码真源：`config/env.ts`。
- LLM 路由真源：`config/llm.ts`。
- 数据模型真源：`server/db/schema/**`、迁移文件与 `plan/reference-schema.md`。
- API 现状真源：`server/app.ts`、`server/routes/**`、OpenAPI registry、`plan/reference-api.md`。
- 运行时配置优先级：`app_settings > .env > 代码默认值`，目标契约见 `plan/reference-schema.md`。
- 计划状态真源：较新的 `plan/step-*`、`plan/reference-*`、`docs/plans/*` 对齐说明。

## 变更流程

1. **识别影响面**：至少标注是否影响 UI、API、数据库、权限、配置、部署、离线内容、agent 调用。
2. **确定依据**：引用本目录和对应 plan/reference 文档。
3. **小步修改**：每次变更保持可审查；避免把样式重设、业务重构和迁移揉在一个 PR。
4. **验证闭环**：按 [11-testing-quality.md](11-testing-quality.md) 完成分层测试。
5. **同步文档**：改了用户可见行为、部署、配置、脚本、状态机、内容产物命名，就必须同步文档。

## 例外机制

任何例外必须写入 PR 描述或计划文档：

- 例外规则编号或文件名。
- 为什么现在不能遵守。
- 风险范围与用户影响。
- 临时防护措施。
- 后续收口日期或触发条件。

禁止使用“时间不够”“先这样”作为唯一理由。

## 冲突处理

- 新规范与 UI/UX 定稿冲突：保留 UI/UX 定稿，先修改规范。
- 新需求与 prebuilt-only 冲突：需求必须重写，不得恢复在线组卷、在线换题或线上内容生成。
- 新实现与已有测试冲突：先判断测试是否仍代表目标契约；如是，改实现；如否，先改计划与测试。
- 文档之间冲突：以更新时间更晚且明确写有“当前对齐说明”的文档为准。

## 评审原则

采用 Google 工程实践中“持续改善代码健康”的思路：评审不追求个人偏好，优先看设计合理性、用户影响、复杂度、测试、命名、注释、文档和一致性。纯样式偏好不得覆盖本目录和 UI/UX 定稿。
